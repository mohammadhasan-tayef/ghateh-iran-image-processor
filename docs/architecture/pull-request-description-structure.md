# Pull Request Description Structure

Every Pull Request description must contain exactly the six level-two sections shown below and preserve their order. This standard defines only why each section exists; it does not prescribe body wording, mandatory fields, validation commands, filling instructions, readiness rules, risk policy, review rules, or merge rules.

## Summary

The Summary section exists to give readers immediate orientation to the proposed change and establish shared high-level context for the rest of the description.

## Why

The Why section exists to make the motivation for opening the Pull Request visible and help readers understand the reason for the proposed responsibility.

## Scope

The Scope section exists to establish a clear boundary around the responsibility represented by the Pull Request, supporting focused review and Sprint traceability.

## Out of Scope

The Out of Scope section exists to make the edges of that responsibility visible and reduce assumptions that adjacent or future work belongs to the Pull Request.

## Validation

The Validation section exists to give verification a consistent, visible location so confidence in the proposed change can be assessed efficiently.

## Risks

The Risks section exists to make uncertainty and potential consequences visible, supporting informed consideration without defining a risk policy.
